# ARL 開発ロードマップ v0.3: リモートGPU環境（RunPod）の統合

このフェーズでは、重い計算を要する機械学習タスクを実行するために、RunPod等のGPUクラウドサーバーとの通信およびDockerベースの環境構築を行います。完全にリモートでの実行・監視ループを確立します。

## 1. Docker環境の定義
- [ ] `docker/base.Dockerfile`: PyTorch, CUDA, 必要なMLライブラリを含んだベースイメージの作成
- [ ] `docker/setup.sh`: コンテナ起動時の初期設定やデータセットのダウンロード等の自動化スクリプト作成
- [ ] ローカル環境でのDockerイメージビルドと、コンテナが想定通り動作するかのテスト

## 2. RunPod統合ツールの実装 (Remote Exec Engine)
- [ ] `tools/runpod_client.py`: RunPod SDKを用いた制御クラスの実装
  - [ ] APIキーとエンドポイントの設定
  - [ ] GPUインスタンス（Pod）の起動リクエスト処理
  - [ ] インスタンスへのファイル（実行コード、データ）転送処理
    - [ ] **拡張**: 単一ファイルではなく、生成されたディレクトリ全体（プロジェクトWorkspace）を同期する仕組み。
  - [ ] Pod上でのコンテナ起動およびスクリプト実行コマンドの発行
  - [ ] 実行ログのポーリング・取得処理
  - [ ] インスタンスの終了・クリーンアップ処理（コスト管理のため極めて重要）

## 3. グラフ実行エンジンの移行
- [ ] v0.2のローカル実行部分（`evaluator.py`内など）を、`runpod_client.py` を用いたリモート実行に置き換える
- [ ] `AgentState` にリモート実行の状態（RunPod ID, Pod status等）を記録するフィールドを追加
- [ ] 長時間実行に対応するための非同期処理（asyncio）の導入、またはポーリングノードのグラフへの追加

## 4. フェールセーフとコスト管理
- [ ] エラー時やタイムアウト時に確実にPodをシャットダウンする仕組みの構築（`try-finally`ブロックや、メインループ外での監視デーモン）
- [ ] Max Retry回数の設定（無限自己修正ループによるクラウド破産を防ぐためのハードリミット）
- [ ] 実行コストの概算とロギング機能の追加

## 5. テストと動作検証
- [ ] `tests/test_runpod.py` (Mock): RunPod APIのモックテスト
- [ ] 実際に安価なGPUインスタンスを起動し、簡単なPyTorch学習スクリプトをリモートで実行・ログ取得・シャットダウンまで行うエンドツーエンドテスト
