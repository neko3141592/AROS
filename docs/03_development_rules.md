# 個人開発ルール & AI協働ガイドライン (Development Rules)

このドキュメントは、ARL (Autonomous Research Loop) プロジェクトを個人で長期的に、かつ圧倒的なスピードで進めるためのルールと、AIエージェント（GitHub Copilot / Cursor / Claude 等）と一緒に開発していく上での心得をまとめたものです。

## 1. 毎日の開発ルーティン・守るべきこと

*   **作業前に「コンテキスト」をセットする**
    *   作業を始める前に、今日はロードマップ（例: `roadmap_v0.1.md`）のどこを進めるのかを確認する。
    *   AIに「今から [タスク名] をやる。関連するファイルはこれとこれ」と宣言してから始める。
*   **「動く状態」で終わる**
    *   一日の終わりに、必ずコードが実行可能（またはテストが通る状態）であることを確認する。エラーのまま放置して翌日に持ち越さない（翌日の自分とAIが文脈を思い出すのに苦労するため）。
*   **迷ったら小さく作る (Small Steps)**
    *   一気に巨大な機能を実装しない。例えば「AIノードの実装」をするなら、まずは「モックで固定文字列を返すノード」を作り、それがグラフ上で正しく遷移するかを確認してから、実際のLLM APIを繋ぐ。
*   **「なぜそうしたか」をドキュメントに残す**
    *   コード内の複雑なロジックや、特定のアーキテクチャを選んだ理由（例：なぜSQLiteか、なぜRedisを併用するか）は、コードのコメントやドキュメントに必ず残す。AIは「現在のコード」しか見えないため、背景の意図を文字にしておくことが重要。

## 2. コミットルール (Git Commit Rules)

個人開発であっても、AIが履歴を読んで背景を理解できるようにPrefixを用いた明確なコミットメッセージをつけること。

### Prefixの種類
*   `feat`: 新機能の追加（例：`feat: RunPodクライアントの初期実装`）
*   `fix`: バグ修正（例：`fix: グラフの無限ループ条件を修正`）
*   `docs`: ドキュメントのみの変更（例：`docs: ロードマップv0.2の更新`）
*   `refactor`: バグ修正や機能追加を含まないコードの書き換え（例：`refactor: Pydanticモデルをディレクトリごとに整理`）
*   `test`: テストコードの追加・修正
*   `chore`: ビルドプロセス、パッケージマネージャなどの変更（例：`chore: langgraphのバージョンアップデート`）
*   `wip`: 作業途中（※翌日に持ち越す場合のみ許容）

### コミットの粒度
*   **「1コミット＝1つの論理的変更」** を徹底する。「Aの機能を実装し、ついでにBのバグも直した」というコミットは避ける。AIが変更の意図をノイズなく読み取れるようにするため。

## 3. AIエージェントとの協働の心得 (Pair-Programming with AI)

AIは**『超優秀だが、プロジェクトの全体像を毎回忘れる記憶喪失のアシスタント』**であるという前提で扱うこと。

1.  **コンテキスト（文脈）が全て**
    *   AIに指示を出すときは、「やりたいこと」だけでなく「関連するファイル」や「システムの前提原則（State-centricであること等）」を明示的に指定（または読み込ませる）すること。
    *   指示の例（悪い）： `plannerノードを作って`
    *   指示の例（良い）： `docs/01_about_this_project.md の仕様と schema/task.py の型定義に基づいて、graph/nodes/planner.py にPlannerノードを実装して。Stateは graph/state.py を用いること。`
2.  **生成されたコードは「レビュー」する**
    *   AIが生成したコードをただコピペして動かさない。AIは時折、存在するはずのないメソッドをでっち上げたり（ハルシネーション）、プロジェクトの設計思想（例えば、全てのやり取りをState経由にするというルール）を無視して直接グローバル変数を書き換えようとすることがある。
    *   「絶対に崩壊しないアーキテクチャ」を守るために、AIの提案が `docs/` 以下の設計書に反していないか、自分がコードリーダーとしてチェックする。
3.  **エラーの丸投げはしない**
    *   エラーが出た際にトレースをそのまま貼り付けるだけでなく、「どこまで合っていて、何が想定外なのか」のリポジトリオーナーとしての仮説を添えると、AIの解決精度が劇的に上がる。
4.  **AIに「リファクタリング」と「ドキュメント化」をやらせる**
    *   動くコードが書けたら、AIに「この機能のDocstringを書いて」「この処理、もっとLangGraphのベストプラクティスに沿ってリファクタリングできる？」と質問し、品質を押し上げる壁打ち相手として活用する。

## 4. このプロジェクト特有の「絶対に守るべき設計ルール」

1.  **State-Centric（状態の一元管理）**
    *   ノード間でデータの直接の受け渡しはしない。必ず `AgentState` を更新し、次のノードは `AgentState` を読み取る。
2.  **型（Pydantic）の絶対視**
    *   データの入出力は必ずPydanticでバリデーションする。辞書(`dict`)の直書きや `kwargs` の乱用は、のちの崩壊の原因になるため避ける。
3.  **副作用はToolに閉じ込める**
    *   外部API通信、ファイル保存、RunPodの起動などの「副作用」はノード内に直接書かず、必ず `tools/` 以下の独立した関数として実装し、ノードからはそれを呼び出すだけにする。これによりテスト可能（モック可能）な状態を保つ。
